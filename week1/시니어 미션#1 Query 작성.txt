1. 미션 1(내가 진행중, 진행 완료한 미션 모아서 보는 쿼리(페이징 포함))에서
정렬 기준을 1순위는 포인트로 2순위는 최신순으로 하여 Cursor기반 페이지네이션을 구현해보세요!

#첫 페이지
SELECT
um.mission_id,
m.name         AS mission_name,
s.name         AS store_name,
m.point        AS mission_point,
um.status
FROM user_mission um
JOIN mission m ON m.id = um.mission_id
JOIN store   s ON s.id = m.store_id
WHERE um.user_id = :user_id
AND um.status IN ('진행중', '성공')
ORDER BY m.point DESC, um.createAt DESC, um.mission_id DESC
LIMIT :limit + 1;     


#첫 페이지 이후부터 
SELECT
  um.mission_id,
  m.name  AS mission_name,
  s.name  AS store_name,
  m.point AS mission_point,
  um.status,
  um.createAt
FROM user_mission um
JOIN mission m ON m.id = um.mission_id
JOIN store   s ON s.id = m.store_id
WHERE um.user_id = :user_id
  AND um.status IN ('진행중', '성공')
  AND (
        m.point < :c_point
     OR (m.point = :c_point AND um.createAt < :c_created_at)
     OR (m.point = :c_point AND um.createAt = :c_created_at AND um.mission_id < :c_mission_id)
  )
ORDER BY m.point DESC, um.createAt DESC, um.mission_id DESC
LIMIT :limit + 1;


2. 다양한 트랜젝션 상태와, 트랜젝션 전파에 대해 조사해주세요!

**트랜잭션의 정의** - DB의 상태를 변경시키는 작업의 단위, ACID 4가지 특징을 가지고 있음

**트랜잭션의 연산**에는 COMMIT(트랜잭션이 성공적으로 수행되었음을 선언하는 연산), ROLLBACK(트랜잭션 수행이 실패했음을 선언하고 작업을 취소하는 연산)이 있음

**트랜잭션의 상태**

![image.png](attachment:8a3755ce-8f73-4094-b734-6beb92611fd2:image.png)

Active(활성)

- 트랜잭션 활동 상태
- 트랜잭션이 실행 중이며 동작 중인 상태를 의미

Partially Committed(부분 완료)

- 트랜잭션의 COMMIT 명령이 도착한 상태
- 트랜잭션의 COMMIT 이전 SQL 문이 수행되고, COMMIT만 남은 상태를 말함

Failed(실패)

- 트랜잭션 실패 상태
- 더이상 트랜잭션이 정상적으로 진행될 수 없는 상태를 말함

Committed(완료)

- 트랜잭션 완료 상태
- 트랜잭션이 정상적으로 완료된 상태를 말함

Aborted(철회)

- 트랜잭션 취소 상태
- 트랜잭션이 취소되고, 트랜잭션실행 이전 데이터로 돌아간 상태를 말함(ROLLBACK연산)

<aside>
💡

❓ **Partially Committed** vs **Committed**

`COMMIT` 요청이 들어오면, **Partially Committed** 상태가 된다.

이후 `COMMIT` 을 문제 없이 수행할 수 있으면 **Committed** 상태로 전이되고, 오류가 발생한다면 **Failed** 상태가 된다.

즉, **Partially Committed** 는 `COMMIT` 요청이 들어왔을 때를 말하며, **Committed** 는 `COMMIT` 을 정상적으로 완료한 상태를 말한다.


**트랜잭션의 전파 방식**

트랜잭션 관리에서 하나의 트랜잭션이 다른 트랜잭션 내에서 호출될 때 그 트랜잭션을 어떻게 처리할지 결정하는 방법을 의미.

Spring에서는 @Transactional의 propagation 옵션으로 설정 가능하다. 

**Propagation.REQUIRED**

- **트랜잭션이 필요하고, 기존 트랜잭션이 있으면 사용하고 없으면 새로 만든다.**
- 메서드가 호출될 때, 호출한 쪽에서 트랜잭션이 설정되어 있지 않다면 새로운 트랜잭션을 시작한다. (새로운 트랜잭션을 생성하여 실행)
- 이미 트랜잭션이 설정된 스코프에서 하위 메서드가 호출되면 상위 메서드의 트랜잭션 내에서 실행된다. (동일한 트랜잭션 내에서 실행)
- 하위 메서드에서 예외가 발생할 경우 해당 트랜잭션은 롤백되며, 이 롤백은 호출한 쪽에도 전파된다.

**Propagation.REQUIRES_NEW**

- **항상 새로운 트랜잭션이 필요하다.**
- 메서드가 호출될 때, 항상 새로운 트랜잭션이 시작된다. (기존 트랜잭션이 있더라도 일시 중지하고, 새로운 트랜잭션을 생성하여 실행)
- 호출한 쪽에 트랜잭션이 설정되어 있더라도, 하위 메서드는 항상 독립적인 트랜잭션 내에서 실행된다. (기존 트랜잭션과 별개로 실행)
- 하위 메서드에서 예외가 발생할 경우, 해당 트랜잭션은 롤백되지만, 상위 트랜잭션에는 영향을 미치지 않는다. (상위 트랜잭션은 계속 진행)



3. 함수 기반 인덱스와 복합 인덱스에 대해 조사하고, 성능상 이점과 단점을 적어주세요!
**함수 기반 인덱스 (Function-Based Index, FBI)**

- **정의:** 컬럼의 값이 아닌, 함수나 연산을 통해 계산된 결과를 인덱스 키로 사용합니다.
- **목적:** 특정 함수나 연산자를 사용하는 쿼리의 성능을 최적화합니다.
- **예시:**
    - `UPPER(first_name)`을 인덱싱하여 대소문자 구분 없이 `first_name`을 검색할 때.
    - `date_part('year', order_date)`처럼 날짜에서 연도를 추출하여 검색할 때.
- **특징:**
    - 인덱싱할 값이 함수나 연산의 결과이므로, 쿼리 작성 시 인덱스와 동일한 함수/연산을 사용해야 합니다.
    - 인덱스의 내부 구조는 일반 B-Tree 인덱스와 동일하며, 인덱스 생성 및 유지관리 비용이 발생합니다.

**복합 인덱스 (Composite Index)**

- **정의:** 두 개 이상의 컬럼을 묶어서 하나의 인덱스를 구성합니다.
- **목적:** 여러 컬럼을 동시에 사용하여 WHERE 절의 AND 조건으로 검색할 때 성능을 향상시킵니다.
- **예시:** `(dept_no, employee_name)`과 같이 부서 번호와 직원 이름을 함께 검색하는 경우에 복합 인덱스를 사용할 수 있습니다.
- **특징:**
    - 단일 컬럼으로는 분포도가 나쁜 경우, 여러 컬럼을 합쳐 좋은 분포도를 가지도록 만들 수 있습니다.
    - 인덱스 컬럼의 순서가 중요하며, 쿼리에서 조건절의 순서와 일치하도록 구성하는 것이 좋습니다.
